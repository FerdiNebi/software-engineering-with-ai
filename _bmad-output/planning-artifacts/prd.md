# Product Requirements Document - End-to-End Software Engineering with AI

**Author:** Ferdi
**Date:** 2026-03-27
**Project Type:** Static documentation site | GitHub Pages | Astro Starlight
**Complexity:** Low (content-driven, not technically complex)
**Project Context:** Greenfield

## Executive Summary

*End-to-End Software Engineering with AI* is a free, static reference site covering the complete software agency delivery lifecycle — from pre-sales through post-launch maintenance — for senior software engineers stepping into consulting and client-facing roles.

Senior engineers know how to build software but rarely know how to run a client engagement end-to-end: scoping calls, discovery workshops, SOW drafting, client sign-off at each phase, and structured handoff. No existing resource covers this. SDLC courses target beginners with abstract methodology. The Pragmatic Engineer covers Big Tech product companies. roadmap.sh covers skill trees. None address the agency/consulting delivery process for engineers who already know how to code.

AI coding tools have made this gap more dangerous. Engineers are expected to move faster and own more scope, but process knowledge hasn't kept pace — resulting in bigger mistakes, faster.

The site is organized as a sidebar navigation with seven phases and 35 sub-sections, each covering what happens, best practices, desired outcomes, and industry approaches. v1 delivers complete traditional (non-AI) content for every page. v2 layers in AI-assisted workflows with battle-tested tool recommendations per phase. All content is generated by LLM agents doing deep research, then reviewed and curated by the author. The site deploys to GitHub Pages as a static build — no runtime APIs, no infrastructure to maintain.

### What Makes This Special

- **The only agency/consulting lifecycle reference for engineers.** Pre-sales, SOW drafting, discovery facilitation, and client handoff are invisible in all existing educational content despite consuming significant senior engineer time. This site makes the invisible visible.
- **Battle-tested, not trend-chasing.** AI tools and workflows are featured only after validation in real client delivery. In a space flooded with hype, this is the practitioner's filter.
- **Process-first, then AI.** v1 establishes the foundational process. v2 layers in AI workflows on top. Most content skips straight to tools without teaching the underlying process — setting engineers up to automate the wrong things.
- **Built by AI, verified by a practitioner.** LLM agents do extensive per-phase research. The author curates for accuracy and real-world applicability. Combines AI breadth with human judgment.
- **Open and community-maintainable.** Markdown content in a public Git repo on GitHub Pages. Anyone can fork, contribute, or self-host.

## User Journeys

### Journey 1: Marcus — The New Consulting Engineer (Primary User, Success Path)

**Marcus** is a senior backend engineer with 8 years at a product company. He's just joined a software consultancy and has his first client kickoff in two weeks. He knows how to architect systems and ship code, but he's never scoped a project, written a SOW, or run a discovery session.

**Opening Scene:** Marcus is staring at a calendar invite: "Discovery Workshop — Client: FinTrack." He has no idea what a discovery workshop involves. He Googles "software consulting discovery process" and finds walls of abstract Agile methodology content that doesn't answer his actual question: *what do I do in that room?*

**Rising Action:** Marcus finds *End-to-End Software Engineering with AI* through a dev community link. He opens the sidebar, sees "Discovery" as a phase, and clicks in. The overview page explains the full discovery phase. He drills into "Requirements Workshops" for specific guidance on what the deliverables are, what best practices look like at real agencies, and what "done" means. He reads the sign-off criteria and realizes he needs to produce a specific set of outputs — not just "gather requirements."

**Climax:** Marcus navigates back to Pre-Sales to understand what should have happened *before* his discovery session — and forward to Requirements & Design to understand what comes *after*. For the first time, he sees the full arc. He's not just preparing for a meeting; he understands where it fits in the delivery lifecycle and what each phase hands off to the next.

**Resolution:** Marcus runs the discovery workshop. It's not perfect, but he knows what he's trying to produce, what questions to ask, and what sign-off looks like. He bookmarks the site and references it at each subsequent phase of the engagement. The process is no longer invisible.

### Journey 2: Priya — The Freelancer Scaling Up (Primary User, Edge Case)

**Priya** is a full-stack developer who's been freelancing for 3 years. She's comfortable building apps solo, but her projects have grown larger — she's now quoting $50K+ engagements and clients are asking for formal proposals, phased delivery plans, and maintenance agreements. She's been winging it and just lost a deal because her proposal "didn't look professional."

**Opening Scene:** Priya searches for "how to write a software project proposal" and finds generic templates that don't reflect how real agencies structure their engagements. She doesn't need a template — she needs to understand the *process* behind the template.

**Rising Action:** She finds the site and goes straight to Pre-Sales & Business Development. The overview gives her the full picture, then she drills into Proposal Writing and SOW & Contract Drafting for specifics. She jumps to Maintenance & Retainer to understand how agencies structure ongoing work — something she's never formalized.

**Climax:** Priya doesn't read the site linearly. She cherry-picks the phases relevant to her immediate problem — Pre-Sales, Discovery, and Maintenance — and skips Development entirely (she already knows how to build). The site's phase-by-phase structure lets her do this without losing context. Each page stands alone while connecting to the whole.

**Resolution:** Priya restructures her next proposal using the pre-sales guidance. She includes a paid discovery phase, a clear scope with sign-off criteria, and a retainer option for post-launch. She wins the deal. The site becomes her reference for the business side of freelancing.

### Journey 3: Ferdi — The Trainer (Trainer Path)

**Ferdi** is preparing a training session for three junior-to-mid engineers joining his consulting team. They can code, but they've never touched the client delivery process. He needs to walk them through the entire lifecycle in a structured way, with enough depth that they leave understanding what each phase requires.

**Opening Scene:** Ferdi opens the site on a shared screen. The sidebar shows the full lifecycle at a glance — all seven phases, ordered top to bottom. He starts at Pre-Sales and tells the group: "This is where every project begins. Let's walk through what happens before a single line of code is written."

**Rising Action:** Ferdi navigates phase by phase, using each overview page as his training outline. "What happens here" gives context. "Best practices" provides the standard to aim for. "Desired outcomes" gives the team a concrete checklist for each phase. He expands Development and drills into specific sub-sections — Code Review, Developer Testing, Secure Development Practices — to go deeper on agency-specific practices.

**Climax:** During the Deployment / Launch section, one of the engineers asks: "What if the client wants changes after launch?" Ferdi navigates to Maintenance & Retainer and shows them how retainer structures work, how incident response is handled, and what the sign-off criteria are for closing an engagement. The site answers the question directly — Ferdi doesn't have to improvise.

**Resolution:** The training session runs for two hours using only the site as the reference material. The engineers leave with a clear mental model of the full delivery lifecycle. Ferdi marks the session successful — the content was structured enough to teach from and detailed enough that he didn't need supplementary materials.

## Technical Requirements

### Architecture

- **Framework:** Astro Starlight — purpose-built docs framework with built-in sidebar navigation, search, and dark mode
- **Content format:** Markdown/MDX files organized by SDLC phase and sub-section
- **Build output:** Static HTML/CSS/JS deployed to GitHub Pages
- **Deployment:** GitHub Actions pipeline triggered on push to main; single-command local build via `npm run build`
- **No runtime dependencies:** zero API calls, zero server-side processing, zero database

### Browser Support

- Modern evergreen browsers: Chrome, Firefox, Safari, Edge (latest two major versions)
- Desktop and mobile rendering required

### Responsive Design

- Content renders cleanly on desktop (primary) and mobile (secondary reference use)
- Sidebar navigation collapses to mobile-friendly menu on small screens (handled by Starlight)
- No custom responsive breakpoints needed beyond framework defaults

### Implementation Considerations

- Content authored as individual Markdown/MDX files — one per phase overview, one per sub-section
- Starlight configuration defines sidebar structure and ordering
- Framework provides built-in search (no external search service needed)
- Dark mode supported out of the box
- No custom JavaScript required for v1
- v2 can leverage MDX and Astro islands for interactive components without architectural changes

## Scoping & Risk

### MVP Strategy

**Approach:** Problem-solving MVP — the product's value is entirely in its content quality and navigability. The technical implementation is deliberately minimal. The MVP succeeds when a senior engineer can navigate to any phase and find complete, actionable guidance.

**Resource Requirements:** Solo developer (Ferdi) + LLM agents for content research and drafting. No team required. Framework setup is minimal; content creation and curation is the primary effort.

### Risk Mitigation

**Content Risks (primary):** LLM-generated content could be generic, inaccurate, or lack practitioner depth. Mitigated by Ferdi's review and curation — no content ships without practitioner validation. Each phase must pass the "would I reference this on a real engagement?" test.

**Technical Risks (minimal):** Astro Starlight is mature and well-documented. GitHub Pages deployment is battle-tested. Mitigated by existing Starlight documentation and templates.

**Scope Risks:** Feature creep into v2 territory before v1 content is complete. Mitigated by hard v1/v2 boundary — v1 ships with zero AI workflow content. The v1 site must stand alone as a complete traditional reference.

## Non-Functional Requirements

### Deployment & Build

- NFR1: Site builds successfully from a clean clone with `npm install && npm run build` — no additional setup steps
- NFR2: Full site build completes in under 60 seconds
- NFR3: Deployment to GitHub Pages achievable via single command or automatic GitHub Actions trigger on push to main

### Content Rendering

- NFR4: All pages render correctly with JavaScript disabled (static HTML output)
- NFR5: Content is readable and navigable on viewports from 320px (mobile) to 2560px (ultrawide) without horizontal scrolling
- NFR6: Dark mode and light mode both render all content legibly with sufficient contrast

### Maintainability

- NFR8: Content files are plain Markdown/MDX with no proprietary format or tooling dependency beyond Astro Starlight
- NFR9: Zero runtime dependencies — all content is pre-built at compile time
