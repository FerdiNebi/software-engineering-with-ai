# Story 1.1: Number the phase labels in the sidebar

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a first-time visitor,
I want the sidebar's 7 phases to be visibly ordered and numbered,
So that I read the sidebar as a lifecycle map, not as a flat menu.

## Acceptance Criteria

1. **Sidebar config carries numeric prefixes (UX-DR3).** In `astro.config.mjs`, each of the 7 phase group `label` strings has a `N. ` prefix matching its lifecycle order:
   - `1. Pre-Sales & Business Development`
   - `2. Discovery`
   - `3. Requirements & Design`
   - `4. Development`
   - `5. QA / Testing`
   - `6. Deployment / Launch`
   - `7. Maintenance & Retainer`

2. **Sub-section labels remain un-prefixed (UX-DR3 — phases only).** No `items[].label` (under any phase) gains a number prefix. The `Overview` label and all other sub-section labels stay exactly as-is.

3. **Phase order in config matches PRD lifecycle order.** The order of the 7 phase groups in `sidebar: [ … ]` is unchanged (already correct in the scaffold). Numbering is the *only* change to the sidebar array.

4. **Numeric prefix is rendered visibly by Starlight on both desktop and mobile**, in both light and dark themes — not stripped, not visually hidden, not collapsed away. Body-text contrast ≥7:1 (UX-DR19); accent-color contrast unchanged.

5. **Existing sidebar behavior preserved**: current-page highlight, current-phase auto-expand (UX-DR4), collapse/expand of other phases, and Pagefind search continue to work unchanged.

6. **Build passes.** `pnpm build` succeeds with no sidebar-config errors and all 43 page routes resolve.

## Dev Notes

**Scope is narrow.** This is a single-file edit (`astro.config.mjs`) that prefixes 7 strings. Do not refactor the sidebar structure, do not introduce a derived constant, do not add a phase registry helper — three similar lines beats premature abstraction. The phase slug registry already exists implicitly in this file (and is documented in `epics.md` lines 115–122); duplicating it elsewhere is out of scope for this story.

**Why no derived constant:** future stories (1.2 `<PhaseList>`, possible 9.8 `<PhaseBadge>`) will need a phase registry, but those stories will introduce it where it's needed. Adding one here would force premature decisions about shape and location. Keep this story to the literal label edit.

**Forbidden in this story:**
- Touching anything in `src/content/docs/`
- Adding any new file (no `src/data/phases.ts`, no helper module)
- Changing slugs (UX-DR3: slugs are immutable post-deploy)
- Touching the `Overview` label or any sub-section label
- Reordering the 7 phase groups
- Adding any custom CSS to render the prefix differently from inline-in-label (Starlight will render the literal string as-is)

**Source tree components to touch:**
- `astro.config.mjs` — only file modified

**Testing standards:**
- No automated test added in this story (the project has no test harness yet — see `spec-site-scaffold.md`). Verification is `pnpm build` + manual visual check per Task 2. A build-time sidebar-label lint is not in scope; pre-launch accessibility audit is Story 9.3.
